# Технический аудит модуля поведения (AL)

Дата: 2026-03-04

## Область аудита

Проверен runtime-контур модуля поведения NPC в `scripts/al_prototype`:
- area orchestration (`al_area_onenter`, `al_area_onexit`, `al_mod_onleave`, `al_area_tick_inc`);
- NPC orchestration (`al_npc_onspawn`, `al_npc_onud`, `al_npc_ondeath`);
- domain-слой (`al_npc_reg_inc`, `al_npc_routes`, `al_npc_acts_inc`, `al_route_cache_inc`).

Важно: стоковые/third-party скрипты не анализировались.

## Краткое заключение

Модуль построен как устойчивый событийный цикл с защитами от рассинхронизации (`al_tick_token`, проверка `al_last_slot`, ресинк при route-failure). Основная архитектура рабочая и предсказуемая для production-эксплуатации.

Обнаружены не критические, но важные эксплуатационные риски: ограниченная наблюдаемость fallback-веток, чувствительность к корректности waypoint-метаданных, и потенциальный event-noise на загруженных локациях.

## Подтверждённые сильные стороны

1. **Единая и безопасная модель lifecycle area**
   - вход/выход игрока и client leave сходятся к одному empty-area handler;
   - stale ticks гасятся через `al_tick_token`, новые тики запускаются только для актуальной «эпохи».

2. **Устойчивая обработка route repeat**
   - `AL_EVT_ROUTE_REPEAT` отбрасывается при неактивном route, пустом route или устаревшем слоте;
   - это защищает NPC от залипания в «прошлом» слоте после ресинков.

3. **Безопасная деградация активности**
   - при невалидной/неподходящей активности модуль сбрасывает активный route и уходит в fallback-поведение;
   - sleep-поток изолирован от лишних повторных route-циклов.

4. **Корректная перерегистрация при смене area**
   - при route-transition NPC снимается со старого registry и регистрируется в новом,
   - после чего форсируется `AL_EVT_RESYNC`.

## Риски и технический долг

1. **Тихие fallback-ветки снижают наблюдаемость контент-ошибок**
   - при нарушении требований активности (не тот tag, разорванная пара) NPC уходит в безопасную ветку без явного сигнала в обычном режиме;
   - без `al_debug=1` билдерам трудно быстро отличить «контент невалиден» от «AI сломан».

2. **Критическая зависимость от качества route metadata**
   - неконсистентные точки маршрута (невалидные location/activity/jump) могут вызывать resync и сбросы маршрута;
   - в runtime это проявляется как «рваное» поведение или частые перезапуски.

3. **Event-noise на массовых сценах**
   - механизм повтора через `AL_EVT_ROUTE_REPEAT` масштабируется линейно по NPC;
   - при высокой плотности NPC увеличивается очередь действий и нагрузка на OnUserDefined.

4. **Регистрационный лимит**
   - registry ограничен `AL_MAX_NPCS`; при переполнении новые участники не попадут в AL-контур;
   - диагностируется только через debug-канал с троттлингом.

## Приоритетный план улучшений

### P1 (высокий приоритет)
- Добавить lightweight телеметрию fallback-событий на area:
  - счётчики `activity_fallback_count`, `route_resync_count`, `route_truncated_count`;
  - сброс счётчиков по `al_tick_token` или по суточному слоту.

### P2 (средний приоритет)
- Подготовить контент-валидатор маршрутов перед релизом:
  - проверка наличия route-tag для активных NPC;
  - проверка `al_activity` и валидности transition-полей;
  - отчёт по точкам, выходящим за ожидаемую area.

### P3 (оптимизация)
- Для перегруженных area сократить среднюю длину маршрутов и число NPC с коротким repeat-циклом;
- оставить `al_debug=1` только на staging и целевых debug-сценах.

## Smoke-сценарий регресса после правок контента

1. Первый вход игрока в area -> NPC unhide + resync.
2. Смена игрового часа через границу слота -> broadcast `AL_EVT_SLOT_*`.
3. Минимум один цикл `AL_EVT_ROUTE_REPEAT` на активном маршруте.
4. Выход последнего игрока -> empty-area cleanup + hide NPC.
5. Повторный вход -> повторный resync без дублей registry.

## Итог

Текущая версия модуля поведения готова к дальнейшей эксплуатации. Наибольшая ценность следующего этапа — не в переписывании core-логики, а в усилении наблюдаемости fallback-путей и в дисциплине контент-валидации маршрутов.
